# Type rendering in the generated docs

The CC API pages under `docs/api/CCs/` are generated by `packages/maintenance/src/docs/generateCCAPIs.ts`. A type-rendering layer (`packages/maintenance/src/docs/renderTypes.ts`) rewrites the raw TypeScript signatures so readers see concrete shapes instead of type constructs. This page documents what is transformed automatically, and which knobs exist when the automatic behavior is wrong.

## Automatic transformations

Signatures of public CC API methods are rewritten as follows:

- **`Pick<T, ...>` / `Omit<T, ...>`** are expanded into object literal shapes. Union targets and generic type parameters keep the named reference and embed the type on the page instead, because expanding a union would drop variant-specific properties.
- **`*CC*Options` types in parameter position** (name matching `/CC\w*Options$/`, directly annotating a parameter) are inlined into an object literal. In other positions (arrays like `ConfigurationCCAPISetOptions[]`, return types) the name is kept and a definition is added to the page's "Related types" section.
- **Compound options parameters** (type aliases that are unions, or intersections containing unions or `AllOrNone<...>`) are rendered as one synthetic overload per variant. `AllOrNone<T>` becomes two variants: one with all properties and one omitting them (for an optional parameter the "none" variant omits the parameter entirely). Hand-written overloads take precedence. Guards: one compound parameter per method, at most 4 variants, all union members must be object literals or named types, bounded total length. When a guard fails the named type is embedded instead.
- **`keyof typeof X`** and **indexed access types** (`SomeCCReport["field"]`) are resolved through the type checker into their concrete types.
- **`extends` clauses** on interfaces are inlined: inherited members are flattened into the definition so readers see the full shape. Bases that cannot be represented as plain members keep the `extends` clause. Add `no-flatten` to the import marker's `with` clause to preserve the `extends` clause deliberately (e.g. `with no-flatten`).
- **Properties typed `T | undefined`** are rendered as optional (`name?: T`).
- Properties that are `@internal`, `@deprecated`, or underscore-prefixed are omitted everywhere.
- **JSDoc is preserved**: member comments survive in embedded definitions and inlined shapes, and a type's own JSDoc summary is rendered as a short description below its "Related types" heading and inside the hover tooltip.

Types that survive these transformations as names are classified by declaration location:

1. Explicit entry in the **type registry** (see below).
2. Declared in `node_modules` (TS lib etc.): ignored.
3. Documented on a hand-written page (auto-detected, see below): linked.
4. Declared in one of our own packages (`packages/*/src/`): definition embedded in the page's "Related types" section, recursively (bounded depth/count). Types that appear in a public signature are documented even when they are only exported from an internal module (e.g. `EndpointCapability`).
5. Anything else (external declarations): counts toward the frequency guard.

## Escape hatches

### The type registry

`typeRegistry` in `packages/maintenance/src/docs/renderTypes.ts` overrides the automatic classification per type name:

- `"expand"` — expand into an object literal shape (used for `Pick`/`Omit`)
- `"embed"` — force a definition in the "Related types" section of pages using it
- `"verbatim"` — leave the name alone: no link, no definition, no warning (e.g. `CommandClasses`, `Bytes`)
- `{ link: "path/to/page" }` — link to an existing docs page without a per-type heading (e.g. `CCAPIs` → `api/CCs/index`)

This is the first classification step, so a registry entry beats every heuristic. An `embed` entry keeps the name in the signature and prevents `*CC*Options` inlining and overload distribution.

### Shared types page and auto-detected link targets

Widely used types belong on `docs/api/shared-types.md`. A `### TypeName` heading on a hand-written page under `docs/api/` becomes a link target automatically when it is directly followed by an import marker (an HTML comment of the form `#import TypeName from "module"`) and/or a `ts` code fence starting with a type declaration. Signatures referencing that name link there. Only PascalCase headings whose slug matches the type name qualify. When the same type is documented on multiple pages, `shared-types.md` wins; other conflicts produce a warning and the link is dropped.

A type earns a place on `shared-types.md` when it is referenced by **at least two CC pages, or at least two hand-written API pages**. Companion types (used inside another entry's definition, like `DurationUnit` within `Duration`) are exempt because they keep that entry self-contained. A type used by a single CC belongs in that CC page's "Related types" section; removing it from `shared-types.md` makes it embed there automatically. The build warns when a `shared-types.md` entry falls below this threshold.

### Frequency guard

A type referenced on **3 or more** CC pages without a documented definition fails the docs build. Fewer references produce a warning. Fix it by adding the type to `shared-types.md` (preferred for `@zwave-js/core` types), exporting it from `@zwave-js/cc` (for CC-specific types), or adding a `verbatim` registry entry to silence deliberately.

### Per-method fallback

A transformation that throws falls back to the method's original source signature and emits a warning naming the CC and method.

## The type-links plugin

`docs/plugins/type-links.js` makes type names in rendered `ts` code fences clickable and shows their definition and JSDoc summary in a hover tooltip. It reads `docs/generated/type-links.json`, which `generateCCAPIs.ts` assembles from auto-detected link targets, per-page embeds, and registry `link` entries. Embeds with conflicting definitions across pages are dropped with a warning. The tooltip copies the tab width from the page's code blocks so its indentation matches the surrounding code.
